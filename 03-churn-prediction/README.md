# 🔄 Проект 3: Прогнозирование оттока (Churn Prediction)

## Задача
Выявить факторы, влияющие на отток пользователей SaaS, и построить простой risk score для раннего предупреждения.

## Данные
- 3,000 пользователей SaaS-платформы
- Признаки: тариф, возраст, сессии, время сессий, обращения в поддержку, дни с последнего входа, неудачные платежи
- Целевая переменная: churned (0/1)

## Статистический метод
**Когортный анализ + корреляционный скоринг** — без ML-библиотек, на чистом Python. Демонстрирует, что даже простая эвристика может давать actionable результаты.

## Что классический аналитик делает 5+ часов
1. Пишет сложные SQL-запросы для агрегации поведения
2. Грузит данные в Python/R + pandas
3. Строит логистическую регрессию или случайный лес
4. Настраивает визуализации
5. Пишет бизнес-рекомендации

## Что делает Augmented Analyst за 30 минут
1. **Агент генерации**: синтетические данные с логичной зависимостью (free-план → больше оттока, давно не заходили → риск)
2. **Агент EDA**: автоматический анализ распределений, корреляций, разницы средних
3. **Агент скоринга**: взвешенная эвристика на основе найденных паттернов
4. **Агент сегментации**: выделение high-risk группы
5. **Евгений**: валидирует логику, проверяет, что risk score не overfit на синтетику

## Ключевые результаты
| Сегмент | Размер | Отток |
|---------|--------|-------|
| Все пользователи | 3,000 | 25.6% |
| Топ-20% по risk score | 600 | **53.5%** |
| План Free | 1,191 | 37.8% |
| План Pro | 612 | 11.6% |
| 0–2 сессий/месяц | 219 | 44.7% |
| >14 дней без входа | — | высокий риск |

## SQL-шаблоны
```sql
-- Базовые метрики по тарифам
SELECT
    plan,
    COUNT(*) as users,
    SUM(churned) as churned_users,
    ROUND(100.0 * SUM(churned) / COUNT(*), 1) as churn_rate,
    AVG(monthly_sessions) as avg_sessions,
    AVG(days_since_last_login) as avg_days_away
FROM churn_dataset
GROUP BY plan
ORDER BY churn_rate DESC;

-- Risk score approximation (SQL)
SELECT
    user_id,
    CASE
        WHEN plan = 'free' THEN 30 WHEN plan = 'basic' THEN 15 ELSE 0 END +
        CASE WHEN monthly_sessions < 3 THEN 25 ELSE 0 END +
        CASE WHEN days_since_last_login > 14 THEN 35 ELSE 0 END +
        CASE WHEN support_tickets > 2 THEN 15 ELSE 0 END +
        CASE WHEN payment_failures > 0 THEN 20 ELSE 0 END
    as risk_score,
    churned
FROM churn_dataset
ORDER BY risk_score DESC
LIMIT 600;
```

## Инсайты
- **Тариф Free** — главный источник оттока (37.8%). Гипотеза: low engagement, нет платёжной привязки.
- **Risk score** увеличивает точность выявления в **2.1x** — с 25.6% до 53.5% в топ-20%.
- **Пороговые правила**: >14 дней без входа + <3 сессий = сигнал для триггерной кампании.
- **Revenue impact**: если реактивировать хотя бы 25% из топ-20% risk score → потенциальная экономия $X/месяц.

## Путь к ML (дальнейшее развитие)
Текущий подход — эвристика. Для production рекомендуется:
1. **Логистическая регрессия** — интерпретируемость + baseline
2. **XGBoost/LightGBM** — если нужна максимальная точность
3. **SHAP values** — объяснение каждого предсказания для бизнеса
4. **A/B тест** — проверить, снижают ли триггерные кампании реальный отток
